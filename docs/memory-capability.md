# DeepResearch 单个任务的 Memory 能力（任务内工作记忆）

> 本文档对应 DeepResearch 链路中的 **memory 能力**（与"精准溯源"、"长程任务"并列），
> 聚焦**单个长程任务内部**的记忆。
> 目标：让 DeepResearch 在**一次用户交互触发、持续十几分钟、需要 agent 多次操作**的
> 单个研究任务中，能够持续维护自己的工作记忆——**不丢目标、不重复劳动、不丢结论**。

---

## 1. 背景与动机

我们关注的场景很具体：**用户单次交互后**，DeepResearch 启动**一个长程任务**，自动完成
资料检索、信息整合与报告生成。这个任务往往要跑十几分钟，agent 要进行**多次操作**——拆解
子问题、反复检索、阅读多篇论文、逐步沉淀结论——并在整个过程中**维护自己的记忆**。

本项目**只聚焦这一个任务内部**的记忆需求：不涉及跨会话复用，也不涉及跨任务的长期知识库。
在单个长程任务里，以"单次对话上下文"为中心的做法会暴露三个结构性问题：

- **上下文窗口会溢出。** 随着检索结果、论文片段、中间结论不断累积，上下文很快被占满。一旦
  超出窗口，早期的检索结果和已读论文就会被"挤掉"，模型在任务后半程开始遗忘自己做过什么。
- **工作会被重复。** 任务很长、要多次重新规划，agent 容易"忘了刚才搜过什么"，于是反复检索
  相同内容，浪费 token 和时间。
- **会丢目标 / 结论散落。** 当前研究问题、子问题列表、提纲草稿，以及从论文中抽取的关键结论
  （claim）、证据，如果只活在对话上下文里，长任务跑到后面就会丢失或难以汇总成报告。

因此，**任务内 memory 能力是支撑单个长程任务"能跑完、跑得连贯"的底层基础**。它不直接负责
"搜得更多"或"溯源更准"，而是负责在这一个任务的十几分钟里，把过程中的状态与知识**留存下来、
组织好、在需要时找回来**。

---

## 2. 核心科学问题

本项目关注的核心问题是：

> **如何让 DeepResearch Agent 在单个长程任务内、在有限上下文窗口下，持续维护自己的工作记忆
> ——保存任务状态、避免重复劳动、留住关键结论？**

这里的"任务内记忆"不是单一一种东西。Agent 在一个长任务的不同步骤需要不同性质的记忆。我们按
**用途**将其划分为三个**facet**（在实现上是同一个存储中的不同 `type` 标签，依靠不同的写入/
读取策略来区分，而非三套独立系统）：

| 记忆 facet | 作用（解决哪个失败模式） | 在单个任务中的例子 |
| --- | --- | --- |
| **工作记忆 (working)** | 防止"丢失当前目标" | 当前研究问题、子问题列表、正在撰写的提纲与草稿、进度快照 |
| **情景记忆 (episodic)** | 防止"重复劳动" | 本任务内已检索过的检索词、已读过的论文、已执行的步骤 |
| **语义记忆 (semantic)** | 防止"结论散落 / 被挤出上下文" | 从论文中抽取的关键 claim、证据句、来源 |

划分依据不是"教科书上有几种"，而是**每一个 facet 各自对应这个长任务里的一种具体失败模式**。

> 这三个 facet 合在一起，就是这个任务里 agent 的"工作记忆"（agent 边做边维护的那份记忆）。

明确**不在范围内**：长期记忆（longterm，跨会话/跨任务复用、用户画像、历史报告知识库）。一旦
任务结束，记忆即可丢弃；下一个任务从空记忆开始。

需要强调：这三个 facet 在实现上**不是三个独立的存储系统**，而是同一个 `MemoryStore` 中带有
不同 `type` 字段的条目，区别体现在**写入策略**（什么值得记、如何去重）与**读取策略**（何时
召回、如何排序）上。

## 3. 主要挑战

### 挑战一：写什么——记忆的写入策略

不是所有信息都值得记。如果把每条检索结果、每段论文原文都塞进记忆，记忆很快就会变成"第二个被
污染的上下文"。难点在于**选择性写入**：

- 什么样的信息值得保留到任务结束（关键结论 vs 背景噪声）；
- 如何**去重**（同一篇论文、同一个 claim、同一次检索动作被多次触发时不应重复存储）；
- 如何在写入时就建立**关联**（claim ↔ 来源论文 ↔ 证据句），方便写报告时直接引用。

### 挑战二：什么时候找回来——记忆的读取策略

记忆的价值在于"**在正确的步骤召回正确的记忆**"。

- 检索相关性（语义相似 / 关键词）只是第一步；
- 还需要结合**新近度**、**记忆 facet**、**当前子任务**进行综合排序；
- 召回后如何**组装进有限的上下文**（top-k 选择、压缩、去冗余），而不是把记忆原样全量塞回——
  这正是对抗上下文溢出的关键。

### 挑战三：记忆的一致性与时效（任务内）

即使在一个任务内，记忆也会不断累积，容易出现：

- **过时**：前半程的临时结论被后续证据推翻，但旧记忆仍被召回；
- **矛盾**：不同论文给出冲突结论，记忆中并存却未标注冲突；
- **漂移**：多轮压缩/摘要后，信息逐步失真。

记忆系统需要能够更新、标注冲突、并在压缩时尽量保真。

### 挑战四：记忆的粒度

同一条信息可以按不同粒度记忆：整篇论文 / 段落 / 单条 claim。粒度太粗不利于精确引用，太细则
碎片化、检索噪声大。需要为不同 facet 选择合适粒度（semantic 以 claim 为单位，episodic 以
"一次动作"为单位，working 以"一次状态快照"为单位）。

### 挑战五：干净的接口

memory 必须以**干净的接口**暴露给流水线：流水线只通过统一的 `MemoryManager` 读写记忆，不直接
触碰底层存储。这样底层实现（内存字典 / 向量库）可以随时替换而不影响调用方；其中 semantic 记忆
里 claim/证据自带 `source`，也为后续"精准溯源"模块在报告里做引用留好了对接点。

## 4. 拟开展的研究内容

围绕上述问题，将任务内 memory 能力设计为一个可插拔、面向单个 DeepResearch 任务的记忆子系统：

```text
DeepResearch 单任务流水线 (规划 → 检索 → 阅读 → 综合 → 写作)
        │  仅通过统一接口读写
        ▼
   MemoryManager  (remember / recall / compress / update_state / get_state)
        │
        ▼
   MemoryStore  (单一存储, 任务内驻留内存 + type 标签: working / episodic / semantic)
        │                         │
        ▼                         ▼
  写入/读取策略 (policies)     检索层 (retrieval: 关键词/向量)
```

整体由**系统管理**：由我们的代码负责抽取、结构化存储、检索召回、组装上下文（类 RAG 的任务内
记忆）。流水线（乃至模型本身，作为工具调用）只通过 `MemoryManager` 这一组方法读写记忆。

### 4.1 记忆 Schema 设计

定义统一的 `MemoryItem`，用一个 `type` 字段区分三个 facet：

```text
MemoryItem:
  id          唯一 ID
  type        working | episodic | semantic
  content     记忆内容（自然语言 / 结构化片段）
  source      来源（论文 ID / URL / 动作 / state）
  tags        主题标签
  keywords    用于检索的关键词
  links       关联（claim ↔ 论文标题 ↔ 证据）
  task_id     所属任务（单任务内一致）
  created_at  写入时间（用于新近度排序与时效判断）
```

### 4.2 写入路径（write path）

- **抽取**：从检索结果 / 论文片段中抽取值得记忆的单元（如一条 claim、一次检索动作）。
- **去重**：写入前按内容 / 来源做相似度去重，避免冗余。
- **关联**：写入时建立 claim ↔ 来源 ↔ 证据 的链接，方便写报告时直接引用来源。
- 不同 `type` 采用不同写入策略（semantic 走"抽取+去重+关联"，episodic 只做轻量追加记录，
  working 以状态快照覆盖式追加）。

### 4.3 读取路径（read path）

- **检索**：按 query 做关键词 / 向量召回（MVP 先用轻量的关键词 + 新近度打分，向量检索作为
  可选项）。
- **排序**：综合相关性、新近度、记忆 facet 进行排序。
- **上下文组装**：选 top-k、必要时压缩，组装成注入提示词的"记忆片段"——只注入精炼后的少量
  内容，其余留在记忆里，从根本上避免上下文溢出。

### 4.4 记忆增强的单任务研究循环

将记忆嵌入这一个任务的每一步：

- **规划阶段**：`update_state` 写下目标/子问题/提纲；`recall` 看是否已有相关结论；
- **检索/阅读阶段**：用 `seen_action` 去重（避免在长任务里重复检索），`remember` 沉淀新发现；
- **综合/写作阶段**：`recall` + `compress`，把分散的 semantic 记忆汇总成报告（带来源引用）；
- **跨步骤连贯**：每完成一组步骤就 `update_state` 快照进度，后续步骤用 `get_state` 重新拿回
  目标，**在十几分钟、几十步里不丢线索**。

### 4.5 评估

在一个多步文献综述任务上，对比有/无任务内记忆时的：

- **重复检索次数**（episodic 去重带来的节省）；
- **上下文占用**（记忆里保留的 findings 总量 vs 实际注入综合步骤的精简量）；
- **目标连贯性**（长任务后半程是否还守得住最初目标/提纲）；
- **报告对证据的覆盖**（最终报告引用了多少任务内沉淀的 claim/来源）。

### 4.6 演进路径

MVP 用"单库 + type 标签、任务内驻留内存"。由于所有访问都经过 `MemoryManager`、且写入/读取
策略已按 facet 拆分，未来可平滑切换为"按后端分库"——例如 semantic 接向量库、episodic 接追加
日志、working 留在内存。届时只是替换存储后端 / 增加路由层，调用方无需改动。

---

## 5. 简要总结

任务内 memory 能力的核心，不是让 DeepResearch "记住更多"，而是让它在**单个长程任务**（一次
交互、十几分钟、多次操作）中保持**连贯、不重复、不丢结论**。

我们从一个**可插拔的统一记忆接口（`MemoryManager`）+ 单一存储 + 三个任务内记忆 facet
（working / episodic / semantic）**切入，由系统管理落地：对内支撑长任务在有限上下文下持续推进，
对外让 semantic 记忆里的 claim/证据自带来源、便于报告引用，并保留向"分库 / 向量化"演进的空间。
